## Module 1: 日常运营 SOP

> 跨境运营的核心不是做得多，而是做得准。每天的固定动作决定了 80% 的运营结果。

### 1.1 Amazon 卖家日常 SOP（亚洲时区）

按时间块组织，适用于中国/东南亚时区卖家运营美国站：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
上午 9:00-11:00 · 数据检查 & 订单处理
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

□ 账户健康检查
  · Account Health Dashboard → 有无新警告/违规
  · 绩效通知 → 有无政策警告、ASIN 下架通知
  · Buyer Messages → 24h 内必须回复（影响 ODR）

□ 订单处理（FBM 卖家）
  · 新订单确认 → 打单发货
  · 异常订单排查（地址不完整、高风险订单）

□ 销售数据快览
  · 昨日销售额 / 订单量 / Session / CVR
  · 与上周同日对比，异常波动 > 20% 需排查
  · BSR 变动（核心 ASIN 排名是否稳定）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
上午 11:00-12:00 · 广告 & 竞品
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

□ 广告数据检查
  · 昨日广告花费 / 销售 / ACOS
  · 异常 Campaign 识别（花费激增但无转化）
  · 预算消耗进度（是否提前花完或花不出去）

□ 竞品快速扫描
  · 核心竞品价格变动（是否降价/加 Coupon）
  · 竞品 Review 动态（新增差评可利用的机会）
  · 新进入者（搜索核心关键词看是否有新面孔）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
下午 14:00-17:00 · Listing 优化 & 运营动作
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

□ Listing 优化（每天专注 1-2 个 ASIN）
  · 关键词排名追踪 → 排名下降的词需要加强广告
  · Search Term Report 分析 → 发现新词、加否定词
  · 主图/A+ Content 优化排期

□ Review 管理
  · 新 Review 查看（负评需要 24h 内回复）
  · Request a Review 操作（订单完成 5-30 天的订单）
  · Vine 计划管理（如有）

□ 库存监控
  · FBA 在途 / 在库 / 可售数量
  · 天数估算：当前库存 ÷ 日均销量 = 可售天数
  · < 14 天预警 → 触发补货流程

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
晚间 20:00-22:00 · 美国白天时段操作（可选）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

□ 广告竞价调整
  · 美国白天是高转化时段，可临时提高竞价
  · 检查 Top-of-Search 展示份额

□ Deal/Coupon 效果监控
  · Lightning Deal / Best Deal 进行中的实时表现
  · Coupon 核销率

□ 客户消息二次检查
  · 确保所有消息已在 24h 内回复
```

## Module 2: 多平台管理流程

> 多平台 ≠ 每个平台照搬一遍。核心是统一管理 + 差异化运营。

### 2.1 平台优先级矩阵

根据收入贡献和增长潜力排序，聚焦资源：

```
评估维度（每项 1-5 分）：

| 平台 | 当前收入 | 增长潜力 | 运营难度(反) | 利润率 | 总分 | 优先级 |
|------|---------|---------|-------------|--------|------|-------|
| {平台A} | ? | ? | ? | ? | ? | P? |
| {平台B} | ? | ? | ? | ? | ? | P? |
| {平台C} | ? | ? | ? | ? | ? | P? |

优先级分配：
  P0（核心平台）：分配 60% 精力 → 深度运营
  P1（增长平台）：分配 25% 精力 → 积极拓展
  P2（测试平台）：分配 15% 精力 → 低成本试水
```

### 2.2 统一 SKU 管理

```
主 SKU 体系设计：

  {品牌缩写}-{品类}-{颜色/规格}-{版本}
  例：MC-BT-BLK-V2 → MarketerClaw 品牌蓝牙耳机黑色第二版

平台映射表：
| 主 SKU | Amazon ASIN | Amazon SKU | Shopee SKU | TikTok SKU |
|--------|-------------|------------|------------|------------|
| MC-BT-BLK-V2 | B0XXXXXX | MC-BT-BLK-V2-FBA | mc_bt_blk_v2 | mc-bt-blk-v2 |

注意：
  · Amazon SKU 不可包含特殊字符
  · Shopee 允许下划线
  · 主 SKU 是内部唯一标识，不受平台限制
```

### 2.3 跨平台定价策略

```
定价原则：
  · 同一产品各平台售价差异 ≤ 15%（避免被消费者比价投诉）
  · 价格差异靠 Coupon/Voucher 实现，不改底价
  · MAP（最低广告价格）全平台统一

定价层级：
  独立站 ≥ Amazon ≥ TikTok Shop ≥ Shopee ≥ Temu

  理由：
    · 独立站无佣金，利润最高，定价可最高
    · Amazon 有品牌信任，支撑较高定价
    · TikTok Shop 内容驱动，用户对价格不敏感
    · Shopee 价格敏感度高，需要有竞争力
    · Temu 低价为王，通常是最低价
```

### 2.4 多平台库存分配

```
库存分配决策树：

总可用库存
├── Amazon FBA（优先保障）
│   └── 按 30 天销量 × 1.5 安全系数分配
├── Shopee 仓（海外仓 / SLS）
│   └── 按 14 天销量分配（Shopee 周转快）
├── TikTok Shop 仓
│   └── 按 14 天销量分配 + 达人带货备货
├── 3PL / 自有仓（FBM + 独立站）
│   └── 剩余库存
└── 备用库存（国内）
    └── 至少保留 1 个月的紧急补货量

⚠️ 注意：
  · 各平台库存不共享时，要防止 A 平台断货 B 平台积压
  · 用统一 ERP 工具管理多平台库存（如 Lingxing、SellerBoard）
```

### 2.5 客户服务 SOP（按平台）

| 场景 | Amazon | Shopee | TikTok Shop |
|------|--------|--------|-------------|
| 回复时限 | 24h（影响 ODR） | 12h（影响 Chat Response Rate） | 24h |
| 退货处理 | FBA 自动退，FBM 需 2 天内处理 | 买家发起后 3 天内响应 | 平台自动处理（全托管）|
| 差评回复 | 不可联系买家改评，可公开回复 | 可通过 Chat 联系买家 | 可公开回复 |
| 退款策略 | 小额直接退款（$10 以下），大额走退货流程 | 视情况部分退款/全额退款 | 跟随平台政策 |
| 投诉升级 | A-to-Z Claim → 尽量在升级前解决 | Shopee Guarantee → 提供证据 | 平台仲裁 |

## Module 4: 新品上架 SOP

> 新品成败的 80% 在上架前就决定了。严格按照时间线执行，不跳步。

### 4.1 新品上架时间线（T-30 到 T+30）

### 4.2 新品上架 Kill Criteria（止损标准）

| 时间节点 | Kill Signal | 决策 |
|---------|-------------|------|
| T+7 | 0 订单 + 0 Session | 检查 Listing 是否被抑制/索引问题 |
| T+14 | 日均 < 1 单，ACOS > 100% | 重新审视选品、定价、Listing 质量 |
| T+30 | 日均 < 2 单，ACOS > 目标值 2x | 降级推广预算，给 2 周观察 |
| T+45 | 无改善迹象，库存 > 90 天可售 | 启动清仓流程（降价 / Outlet Deal / 移除）|
| T+60 | 仍无起色 | 彻底放弃，清理库存，复盘失败原因 |

### 5.1 组织架构模板（按规模）

**Solo Seller（1人）：**

```
你 = 选品 + 采购 + 运营 + 广告 + 客服 + 物流

时间分配建议：
  运营 & 广告：40%
  选品 & 采购：25%
  客服 & 售后：15%
  财务 & 数据：10%
  学习 & 规划：10%

生存法则：
  · 聚焦 1 个平台（Amazon US 优先）
  · SKU 数量 ≤ 10
  · 能外包就外包（摄影、物流交给服务商）
  · 用工具替代人力（ERP、自动邮件、广告自动化）
```

**小团队（2-5 人）：**

```
角色分配：

负责人（你）
├── 运营专员 A：Listing 优化 + 广告投放
├── 运营专员 B：客服 + Review 管理 + 售后

  · 每周一 30 分钟周会（上周复盘 + 本周重点）
```

### 5.3 KPI 框架（按角色）

**运营岗：**

| KPI | 计算方式 | 目标参考值 |
|-----|---------|-----------|
| 月销售额 | 负责 SKU 总收入 | 环比增长 > 10% |
| BSR 排名 | 核心 ASIN 品类排名 | 维持或上升 |
| 新品成功率 | 30天达标产品 / 总上新数 | > 60% |
| Listing 质量分 | Amazon Listing Quality Score | > 8/10 |
| 账户健康 | Account Health Rating | Good |

**广告岗：**

| KPI | 计算方式 | 目标参考值 |
|-----|---------|-----------|
| ACOS | 广告花费 / 广告销售 | < 品类均值 |
| TACoS | 广告花费 / 总销售 | < 12% |
| 新词发现 | 每月发现的有效新关键词数 | > 20 |
| 广告销售占比 | 广告销售 / 总销售 | 逐月下降趋势 |

**供应链岗：**

| KPI | 计算方式 | 目标参考值 |
|-----|---------|-----------|
| 断货率 | 断货 SKU 天数 / 总 SKU 天数 | < 3% |
| IPI 分数 | Amazon IPI Score | > 500 |
| 物流成本占比 | 物流总费 / 总收入 | < 15% |
| 采购交期达成率 | 准时到货批次 / 总批次 | > 90% |

### 5.4 招聘优先级指南

```
判断下一个该招什么人：

瓶颈在哪？ → 对应招谁

"我每天忙着回消息处理退货"
  → 招客服专员（最先释放创始人精力）

"广告花钱太多但我不会优化"
  → 招广告投手（或先外包给代运营测试）

"总是断货 / 物流出问题"
  → 招采购物流专员

"有产品但 Listing 不行 / 不会做图"
  → 招运营 + 美工（或外包美工）

"想扩平台但没人管"
  → 招平台运营（先让核心平台的人培训）

通用建议：
  · 先外包再招聘（验证需求是否持续）
  · 一次只招一个人（充分磨合）
  · 必须有标准 SOP 才能招人（否则新人无法接手）
```

## 执行步骤

### Step 1: 了解团队现状

收集用户的团队规模、运营平台、当前痛点。如未提供，追问：
- 你的团队多少人？各自负责什么？
- 运营哪些平台？哪个是主力？
- 最大的效率瓶颈在哪？

### Step 2: 选择模块

根据用户需求选择对应模块执行：
- "日常运营怎么安排" → Module 1
- "多平台怎么管" → Module 2
- "帮我做个复盘模板" → Module 3
- "新品上架流程" → Module 4
- "团队怎么分工" → Module 5
- "给我一个运营日历" → Module 6
- "全部都要" → 按 Module 1→6 顺序执行

### Step 3: 定制化输出

根据用户的平台、团队规模、具体产品定制 SOP 内容。不要输出与用户无关的平台内容。

### Step 4: 生成可执行文档

输出格式化的 SOP / 复盘模板 / 日历，用户可以直接复制使用。

### Step 5: 关联其他技能

在 SOP 中标注与其他技能的衔接点：
- 选品环节 → mc-selection
- 成本计算 → mc-cost
- Listing 优化 → mc-listing
- 广告策略 → mc-ads
- 合规检查 → mc-compliance
- 竞品分析 → mc-compete / mc-monitor
- 物流决策 → mc-logistics

### Step 6: 交付

按交付规范输出摘要卡片。

- 9:00 Amazon 账户健康 + 销售数据
- 10:00 广告数据检查 + 竞价调整
- 14:00 Listing 优化任务
- 16:00 Search Term Report 分析

## 常见陷阱

1. **SOP 写了不用** — 最好的 SOP 是最简单的。一页纸 > 十页文档。贴在工位上比存在网盘里有用。

2. **所有平台同等投入** — 新手最常犯的错。3 个平台各投 33% 精力 = 3 个平台都做不好。先把一个平台做到稳定盈利再扩。

3. **不做数据复盘** — "感觉还行"是最危险的。每周 2 小时复盘，比每天多干 2 小时更有效。

4. **新品不设 Kill Line** — 沉没成本谬误。设定明确的止损标准，到了就执行，不要"再给它一次机会"。

5. **团队没有 SOP 就招人** — 新人来了不知道干什么，老板天天教 = 更忙。先写 SOP，再招人。

6. **忽略时区差异** — 中国晚上 8-10 点是美国白天高峰。如果这个时段不做广告调整，会错过最佳优化窗口。

7. **只看收入不看利润** — 月销 $50K 但净利 $2K 的产品不如月销 $10K 净利 $3K 的产品。用 mc-cost 定期审计每个 SKU 的真实利润。

## 核心问题

- 我们的运营时间花在最有杠杆的事情上了吗？
- 团队中最大的瓶颈岗位是什么？
- 如果明天有人离职，其他人能接手吗？
- 我们的复盘是否驱动了实际的运营改进？
- 新品成功率是多少？失败产品有没有共性？
- 库存断货和积压的频率是否在下降？
- 我们的 SOP 上次更新是什么时候？还适用吗？
- 每个人是否清楚自己的 KPI 和考核标准？

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

   · 团队：{团队规模} · 平台：{平台组合}
```

## 交互规则

- 用户可以只要某个模块（如"只给我复盘模板" / "新品上架 checklist"）
- 根据用户的平台组合裁剪内容，不输出无关平台的 SOP
- 根据团队规模调整复杂度（Solo 要简单，15 人团队要精细）
- SOP 模板中的 {变量} 用用户提供的实际信息填充
- 复盘模板中的 KPI 目标值根据用户的品类和阶段调整
