## Module 3: 需求验证

### 3.1 关键词簇分析

将品类相关关键词按搜索意图分簇，验证需求的真实结构：

**关键词簇分类框架**：

| 簇类型 | 定义 | 示例（以便携榨汁杯为例） | 选品意义 |
|--------|------|------------------------|---------|
| 核心需求簇 | 品类通用词，代表基础需求 | portable blender, personal blender | 需求基本面 |
| 功能需求簇 | 特定功能/属性需求 | usb rechargeable blender, self-cleaning blender | 差异化方向 |
| 场景需求簇 | 使用场景驱动 | blender for smoothies, travel blender for gym | 定位方向 |
| 人群需求簇 | 特定人群 | blender for baby food, blender for elderly | 细分机会 |
| 痛点需求簇 | 解决特定问题 | quiet blender, easy to clean blender | 产品改进方向 |
| 对比需求簇 | 与其他方案对比 | blender vs juicer, portable blender vs ninja | 竞品关系 |

**关键词簇分析表**：

| 关键词簇 | 代表词 | 月总搜索量 | 词数量 | 搜索量趋势 | 需求确定度 |
|---------|-------|----------|--------|----------|-----------|
| 核心需求簇 | | | | | ★★★★★ |
| 功能需求簇 | | | | | ★★★★ |
| 场景需求簇 | | | | | ★★★★ |
| 人群需求簇 | | | | | ★★★ |
| 痛点需求簇 | | | | | ★★★★★ |
| 对比需求簇 | | | | | ★★★ |

**需求集中度分析**：

```
需求集中度 = 核心需求簇搜索量 ÷ 全部簇总搜索量

> 70% → 需求集中（产品形态明确，但差异化空间小）
40%-70% → 需求适度分散（有细分机会）
< 40% → 需求高度分散（品类可能太宽泛，需聚焦子品类）
```

### 3.2 Review 挖掘：发现未满足需求

**Review 分析框架**：

从品类 Top 5-10 卖家的 Review 中提取以下信息：

| 分析维度 | 数据源 | 产出 |
|---------|-------|------|
| 高频好评词 | 4-5 星 Review | 用户最看重的价值点 |
| 高频差评词 | 1-3 星 Review | 现有产品的核心缺陷 |
| 功能期望 | "I wish..." / "If only..." | 未满足的功能需求 |
| 质量抱怨 | "broke" / "stopped working" | 质量改进方向 |
| 使用场景 | "I use it for..." | 真实使用场景画像 |
| 竞品对比 | "compared to..." / "switched from..." | 竞品优劣对比 |

**Review 挖掘输出模板**：


### 3.3 "用户之声"提取

超越 Review，从更广泛的渠道捕捉用户真实需求：

**数据源与提取方法**：

| 渠道 | 搜索方式 | 关注什么 |
|------|---------|---------|
| Amazon Q&A | 品类 Top ASIN 的 Q&A 区 | 购买前的疑虑和关注点 |
| Reddit | r/BuyItForLife、品类相关 subreddit | 深度使用体验、长期痛点 |
| YouTube | 品类评测视频评论区 | 购买决策因素、使用场景 |
| TikTok | 品类相关话题标签 | 年轻用户的使用场景和态度 |
| Facebook Groups | 品类爱好者/用户社群 | 核心用户的深层需求 |
| Google "People Also Ask" | 品类关键词搜索结果 | 用户最常见的疑问 |

**用户需求层次分析**（Kano 模型应用）：

| 需求层次 | 定义 | 选品意义 | 示例 |
|---------|------|---------|------|
| 基本需求 | 没有就差评，有了不加分 | 产品底线，必须满足 | 充电线够长、不漏水 |
| 期望需求 | 做得好加分，做不好扣分 | 核心竞争力来源 | 续航长、声音小、清洗方便 |
| 兴奋需求 | 有了惊喜，没有不扣分 | 差异化亮点 | 自带食谱APP、创新配色 |
| 无差异需求 | 有没有用户都无感 | 避免浪费成本 | 华而不实的附加功能 |
| 反向需求 | 加了反而扣分 | 必须避开 | 过于复杂的操作、不必要的附件 |

### 3.4 跨平台需求交叉验证

单一平台数据有偏差，用跨平台数据交叉验证需求的真实性：

**交叉验证矩阵**：

| 验证维度 | Amazon 数据 | Google Trends | 社交媒体 | 综合判断 |
|---------|-----------|--------------|---------|---------|
| 需求存在性 | 搜索量 >X | 搜索兴趣稳定/上升 | 话题讨论量增长 | 确认/存疑/否定 |
| 需求趋势 | 搜索量环比 | 5 年趋势线 | 话题热度变化 | 上升/稳定/下降 |
| 细分机会 | 长尾词分布 | 关联查询 | 用户讨论焦点 | 有/无明确细分 |
| 地域差异 | 各站点表现 | 区域热度分布 | 不同市场讨论差异 | 有/无地域差异 |

**跨平台一致性评分**：

```
3/3 平台验证一致 → 高置信度（★★★★★）— 需求确定
2/3 平台验证一致 → 中置信度（★★★）— 需要进一步调查
1/3 平台验证一致 → 低置信度（★）— 需求存疑，谨慎投入
0/3 平台验证一致 → 需求不成立 — 建议放弃
```
